Git as the Control Plane: GitOps Principles That Survive Production

GitOps is not “Kubernetes plus a Git repo.” It is an operating model: declare how the world should look, store that declaration where humans collaborate, and let automation reconcile reality with the declaration—continuously, audibly, and safely.

In short

Treat Git as the single source of truth for desired state, keep that state declarative, let controllers pull and reconcile changes, and wire governance (reviews, policy, secrets) around the same workflows you already use for code.

What GitOps is (and is not)

GitOps describes a way to operate cloud-native systems—often Kubernetes, but the ideas travel—where the approved configuration of a system lives in version control and automated agents apply it. The name caught on after Weaveworks popularized the pattern; implementations include Flux, Argo CD, and similar controllers.

GitOps is not a replacement for CI/CD, observability, or security. It sits beside them: continuous delivery still builds and tests artifacts; GitOps focuses on which version of infrastructure and workloads should run, how that decision is recorded, and how clusters stay aligned with that record.

1. Declarative desired state

Imperative scripts (“run these fifty commands in order”) are hard to review and brittle when order or timing changes. GitOps favors declarative manifests: YAML, Helm, Kustomize, Terraform plans committed as code—artifacts that describe what should exist, not every low-level step to get there.

Declarative models make diff-based review possible: a pull request shows how the world will differ after merge. That same diff becomes evidence for auditors and on-call engineers retracing an incident.

2. Version control as the system of record

Git is the source of truth for what is allowed to run—not a secondary mirror of “what someone clicked in a console last Tuesday.” Commits attach identity, time, and rationale (via messages and linked tickets). Branches and tags model promotion paths: dev → staging → production.

This principle breaks if teams still “fix” production by hand without backporting changes to Git. Drift between live systems and the repo erodes trust; GitOps works best when the rule is: if it is not in Git, it does not belong in the cluster.

3. Automated delivery through pull-based reconciliation

In a typical GitOps setup, a controller inside the cluster watches the Git repository (or an OCI registry of packaged manifests) and pulls the desired state. It compares cluster reality to the declaration and applies updates. That pull-based model avoids handing long-lived credentials from CI into the cluster for every deploy; the cluster asserts what it should fetch.

Push pipelines still matter for building images and running tests; GitOps usually takes over at the boundary of “this release is approved—now make the platform match.”

4. Continuous reconciliation and drift detection

Reconciliation is not a one-time deploy. The controller re-queues work so that manual changes—or failed partial applies—are detected and corrected according to policy. In practice you choose whether unauthorized drift is automatically reverted, flagged, or blocked by admission policy.

This is where GitOps intersects SRE: the system’s desired state is a contract; the controller’s metrics and alerts expose when the contract cannot be met (sync failures, resource contention, bad manifests).

5. Everything reviewable: the same PR culture as application code

Infrastructure changes flow through merge requests with reviewers, CI checks, and optional policy gates. That yields a social layer on top of automation: two keys for production, separation of duties, or platform-team oversight can map cleanly onto branch protection.

Edge case: emergency break-glass exists in every organization; the GitOps principle is to record the outcome—revert or promote a hotfix through the repo quickly—so the system of record catches up with reality.

6. Security and secrets: defaults, not afterthoughts

GitOps amplifies good hygiene and punishes bad secrets handling.

  • No long-lived secrets in plain Git. Use sealed secrets, external secret operators, cloud KMS references, or short-lived identity federation.
  • RBAC and tenancy limit which namespaces or clusters a controller may touch; blast radius should match team boundaries.
  • Supply chain: sign commits or artifacts where policy requires it; pin Helm charts and container images to digests when stability matters.
  • Policy-as-code (OPA, Kyverno, admission webhooks) encodes constraints—“no privileged containers,” “only approved registries”—at apply time.

7. Observability: sync status is part of the product

A GitOps rollout is not done when CI turns green; it is done when the controller reports healthy sync and workloads meet SLOs. Dashboards and alerts should cover application metrics and sync lag, prune failures, and dependency errors between environments.

Remediation paths should be obvious: “which commit is live?,” “which manifest failed validation?,” “which dependency could not be fetched?”—answers your Git URL, commit SHA, and controller UI or metrics should make visible.

Common pitfalls (so principles become practice)

  • Treating GitOps as only a tool install. Without declarative discipline and review culture, you automate confusion faster.
  • Monorepo sprawl without ownership. Define who approves which paths; use CODEOWNERS or subtree policies.
  • Ignoring bootstrap and cluster lifecycle. The cluster and the GitOps agent itself need an intentional birth story—often a small, audited seed.
  • Forklift “GitOps” over SSH kubectl from CI. That can work, but it often forfeits pull-based reconciliation and drift semantics.

How this fits next to DevOps and platform engineering

GitOps is a concrete pattern inside the broader DevOps aim: short feedback loops, shared ownership, and measurable delivery. Platform teams often offer a paved road: golden templates, cluster blueprints, and guardrails so product teams get GitOps benefits without each squad becoming kubectl experts overnight.
