The Container Runtime Interface (CRI): How Kubernetes Actually Runs Containers

Kubernetes schedules Pods; something on each node must create namespaces, pull images, start processes, and report status. That “something” is not the API server—it is the kubelet talking to a container runtime through the Container Runtime Interface (CRI). CRI is the contract that let Kubernetes shed its early Docker dependency and standardize on OCI-compliant engines like containerd and CRI-O.

In short

CRI is a gRPC API between kubelet and a runtime. The runtime implements pod sandboxes and containers; underneath, OCI runtimes (typically runc) create the actual Linux namespaces and cgroups. Build images with Docker or BuildKit; run them in the cluster via CRI—same image format, different control path.

The problem CRI solved

Early Kubernetes embedded Docker knowledge deep in the kubelet. Docker was both a developer tool and a low-level runtime, but its API was not designed for Kubernetes’ pod model (multiple containers sharing network and IPC, pause containers, granular lifecycle). Every Kubernetes release risked chasing Docker’s API surface.

In 2016 the project introduced CRI: a narrow, Kubernetes-owned gRPC interface. Runtimes implement CRI; kubelet calls it. Docker itself never implemented CRI natively—clusters used a dockershim adapter inside kubelet until it was removed in Kubernetes 1.24 (2022). Today’s supported path is: kubelet → CRI → containerd or CRI-O → OCI runtime (runc, crun, etc.).

If you understand containers from the host up (namespaces, cgroups, images), see Docker—the hidden side. For where kubelet sits in the cluster, see Kubernetes architecture in simple terms.

Layered model: who does what

Confusion often comes from stacking three different “runtimes” in one sentence. Separating layers keeps troubleshooting sane:

  • Orchestrator (Kubernetes) — Decides which Pod runs where, health checks, volumes, secrets projected into the pod spec. The kubelet is the node agent that enforces that spec locally.
  • CRI implementation (containerd, CRI-O) — Speaks CRI to kubelet. Manages image pulls, pod sandboxes, container create/start/stop, logging endpoints, and status. This is what people mean by “container runtime” in K8s docs.
  • OCI runtime (runc, …) — Takes an OCI bundle (config + rootfs) and creates the isolated process on Linux. Fast, minimal, no cluster awareness.

Images follow the OCI image spec—layers, manifest, config—so an image built with Docker Desktop or BuildKit is the same artifact containerd pulls when kubelet asks for it.

CRI services and the pod sandbox

CRI defines two gRPC services (protobuf in the Kubernetes repo, versioned as runtime.v1):

  • RuntimeService — Pod sandboxes, containers, exec/attach/port-forward, status, and (where supported) checkpoint/restore hooks.
  • ImageService — Pull, list, and remove images; map image names to filesystem references the runtime uses.

Kubernetes does not ask the runtime to “run a Deployment.” It asks for a pod sandbox first, then one or more containers inside that sandbox.

Pod sandbox — A logical environment with its own network namespace (and typically IPC/UTS sharing policy). The famous pause container (registry.k8s.io/pause or similar) often holds the sandbox alive while application containers start and exit. All containers in a Pod share the sandbox’s network namespace—hence localhost between sidecars works.

Typical sequence when kubelet syncs a Pod:

  1. RunPodSandbox — Create sandbox; return sandbox ID.
  2. For each container: CreateContainer (image, command, mounts, security context mapped to Linux options) → StartContainer.
  3. Periodic ContainerStatus / ListContainers for readiness and restart decisions.
  4. On teardown: StopContainer → RemoveContainer → StopPodSandbox → RemovePodSandbox.

Image pulls happen via ImageService.PullImage (with credentials from kubelet-supplied secrets), often before sandbox creation. Pull policy IfNotPresent / Always is enforced at the kubelet layer; CRI executes the pull.

What kubelet sends: from Pod spec to CRI config

The API server stores a Pod manifest; the kubelet converts it into CRI messages. Important mappings operators feel in production:

  • Resources — CPU/memory limits become cgroup constraints applied when the OCI spec is built.
  • Security context — runAsUser, runAsNonRoot, readOnlyRootFilesystem, capabilities, seccompProfile, selinuxOptions translate into Linux security settings on the container process.
  • Mounts — ConfigMaps, Secrets, emptyDir, hostPath, and CSI volumes become bind mounts in the container filesystem namespace.
  • Environment and command — Literal env, downward API, and secret-backed env vars are resolved by kubelet before CreateContainer.
  • Probes — HTTP/TCP/exec probes are executed by kubelet (not CRI), but they depend on container status and network namespace from the sandbox.

When something fails at “container won’t start,” the break is often in this translation—wrong mount, forbidden capability, or an image pull auth issue—not in the scheduler.

containerd and the shim architecture

containerd is the default CRI backend on many distributions (EKS, GKE, AKS, kubeadm, kind, minikube). Internally it is more than CRI:

  • containerd — Daemon exposing CRI (when configured), image store, snapshotters (overlayfs, etc.), and metadata.
  • containerd-shim — Per-container supervisor; keeps I/O and exit status after containerd restarts. Version 2 shims speak ttrpc and work with runc v2.
  • runc — OCI-compliant low-level runtime that creates namespaces and cgroups.

That separation matters for upgrades: restarting containerd should not kill all workloads because shims hold the process relationship. It also matters for debugging: you may see containerd-shim-runc-v2 in ps alongside your app.

Docker Desktop and standalone Docker Engine still use containerd under the hood for container execution in modern versions—but docker run goes through Docker’s API, not CRI. Only kubelet uses CRI on the node.

CRI-O: Kubernetes-native, OCI-first

CRI-O was built by Red Hat and the community specifically to implement CRI without carrying Docker’s historical baggage. It is lean, defaults to runc (or alternative OCI runtimes), and integrates tightly with Kubernetes release semantics—popular on OpenShift and many RHEL/Fedora/CentOS Stream flows.

Choosing CRI-O vs containerd is often a distribution decision, not a capability lottery: both implement the same CRI contracts Kubernetes tests. Differences show up in tooling (admin familiarity), storage/snapshot plugins, and operational packaging—not in whether Pods work.

dockershim removal and what still uses Docker

Until 1.24, you could point kubelet at Docker via dockershim. That path is gone. You should assume:

  • Inside the cluster — containerd or CRI-O (or another certified CRI implementation).
  • On developer laptops — Docker or Podman for build/test; images push to a registry; the cluster pulls by reference.
  • CI — docker buildx, Kaniko, BuildKit, or cloud-native builders—all produce OCI images consumable via CRI.

“We use Docker in prod” for Kubernetes usually means “we use Docker to build” or “we use Docker on the VM for legacy scripts,” not “kubelet talks to dockerd.”

CRI vs CNI vs CSI — three interfaces, one node

Nodes integrate Kubernetes through several acronyms; mixing them causes muddled incident notes:

  • CRI — Containers and images (this article).
  • CNI (Container Network Interface) — Plugs network interfaces into the pod sandbox (Calico, Cilium, Flannel, AWS VPC CNI, etc.). kubelet calls CNI after the sandbox exists.
  • CSI (Container Storage Interface) — Attaches and mounts volumes; kubelet coordinates with node plugins and API objects (PV/PVC).

A Pod stuck in ContainerCreating might be CRI (image pull, mount), CNI (IP allocation), or CSI (volume attach)—check events and component logs with that split in mind.

Observability and debugging from the node

kubectl speaks to the API server, not to CRI. For node-local truth, use crictl (CRI-compatible CLI) with the same socket kubelet uses:

# Typical socket paths (vary by distro)
# containerd: /run/containerd/containerd.sock
# CRI-O:      /var/run/crio/crio.sock

export CONTAINER_RUNTIME_ENDPOINT=unix:///run/containerd/containerd.sock

crictl pods
crictl ps
crictl images
crictl inspect <container-id>
crictl logs <container-id>

Compare with ctr -n k8s.io containers list for containerd’s native namespace—useful when CRI and containerd metadata disagree after a partial failure.

kubelet logs (journalctl -u kubelet) show CRI RPC failures, image pull backoffs, and sandbox creation errors. Runtime logs (containerd / crio) add snapshotter and shim detail.

Security and operations implications

  • Socket access is root-equivalent. Anyone who can call the CRI socket on a node can start privileged workloads. Lock down SSH, protect break-glass, and treat node compromise as cluster compromise.
  • Image pull secrets flow kubelet → CRI → registry. Misconfigured IAM, expired tokens, or rate limits surface as ImagePullBackOff in the API, rooted in CRI pull errors.
  • User namespaces and rootless — Emerging Kubernetes features push identity mapping closer to the OCI runtime; CRI implementations must expose the right hooks. Stay current on release notes if you harden with user namespaces.
  • Sandboxed runtimes — gVisor (runsc) and Kata Containers present different OCI-compatible runtimes selectable in containerd/CRI-O config for stronger isolation at a performance cost—not a different Kubernetes API.
  • Garbage collection — kubelet evicts unused images and dead containers per thresholds; containerd/CRI-O also GC storage. Disk pressure on nodes is often image layer accumulation—watch crictl images and node allocatable/ephemeral storage.

Runtime comparison (practical, not religious)

Engine Role Typical use
containerd CRI + image/store + shims Default on major clouds and kind/minikube; broad docs and tooling
CRI-O CRI-only daemon, OCI-focused OpenShift/RHEL ecosystems; minimal surface aligned to K8s releases
runc OCI low-level runtime Actually creates containers; swapped for crun, runsc where configured
Docker Engine Developer/CI API; embeds containerd Build and local run; not kubelet’s CRI target since dockershim removal

Failure modes you will see in production

  • ErrImagePull / ImagePullBackOff — Registry auth, wrong tag, air-gapped cluster without mirror, or platform arch mismatch (amd64 image on arm64 node).
  • CrashLoopBackOff — App exits; CRI reports exit code; kubelet backs off restarts. Root cause is usually application or probe, but check crictl logs before blaming the network.
  • CreateContainerConfigError — Secret/ConfigMap missing, invalid subPath, forbidden sysctl.
  • Sandbox not ready — CNI failed to assign an IP; looks like CRI from the outside but kubelet events mention network plugin timeouts.
  • Disk pressure evictions — kubelet evicts Pods; image GC and log rotation on the node are part of the story.

How CRI fits the rest of your platform story

CRI is the mechanical layer under the declarative world described in GitOps principles and the access controls in Kubernetes cluster RBAC. Git declares an image digest; the scheduler places the Pod; kubelet drives CRI until the sandbox and containers match the spec. None of that replaces observability, policy admission, or sane resource requests—but when containers misbehave, knowing this stack separates “API problem” from “node execution problem.”
